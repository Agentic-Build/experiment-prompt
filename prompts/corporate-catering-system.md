# 企業訂餐系統

## Background

為了解決員工長期於廠內駐點餐廳用餐，而產生用餐疲勞的問題。公司計畫額外引入外部商家提供的外送服務，豐富員工的用餐選擇。然而，幾萬人的公司若手動管理外部訂單，將面臨：餐廳資訊混亂、份數統計出錯、領餐現場失控等管理困境。本系統旨在提供一個自動化平台，讓外部商家能自主管理菜單與供應量，並讓員工得以提前預購餐點。

---

## Target Audience

* **員工**：厭倦員餐、想提前規劃未來數天午餐，並在指定地點領取外送美食的人
* **合作商家**：能自行更新菜單、設定每日最大供應份數（限量），並根據匯總單據進行備餐與配送的人
* **福委會管理員**：負責審核外部商家入駐，並透過系統自動對帳扣款，不需介入每日訂單細節

---

## Interview Synthesis

### 核心訪談結論（作為 PRD 需求來源）

* 本產品不是追求「功能數量」，而是先解決三方在高規模情境下的流程斷點：訂單資訊混亂、供應量失準、現場領餐低效率、月結扣款與例外處理負擔過高。
* 需求需區分「真實觀察」與「推測」，避免把未驗證想法直接當成核心功能。
* MVP 需優先確保流程可跑通，再逐步導入推薦、進階儀表板等加值能力。

### 角色需求整理（訪談摘要）

1. 員工

* 需要多日（3/5/7 天待驗證）預購能力，核心動機是省決策時間、降低熱門餐點售完風險。
* 領餐流程重點是快、準、不尷尬，驗證方式需在員工證 / QR Code / 報工號姓名中選擇現場最可行方案。
* 付款方式可接受薪資代扣，但必須具備可查詢、可申訴、可追蹤異常處理狀態的透明機制。
* 推薦非絕對核心；搜尋與篩選（口味、價格、健康、是否可送達）通常優先於 AI 推薦。

2. 合作商家

* 後台重點是「快速上架與改量、降低出錯」，需支援菜單複製、每日份數設定、臨時缺貨處理。
* 截單時間與備餐節奏是商家穩定出餐關鍵，必須定義可接受的最晚異動時間。
* 多廠區配送需要清楚的分裝與配送輸出（單張標籤、分區總表、配送籃清單），以降低漏餐與送錯風險。
* 特殊備註需可控，建議以固定選項為主，限制自由文字，避免過度客製造成出餐風險。

3. 福委會管理員

* 商家審核需完整文件生命週期（上傳、審核、到期提醒、停權）與操作留痕。
* 商家 x 廠區服務範圍需可配置且可快速調整，避免員工看到不可配送店家。
* 月結扣款應以「批次自動化 + 例外清單處理」為核心，支援符合 HR 格式匯出與鎖帳機制。
* 需具備異常預警與治理能力（文件到期、準時率下降、滿意度過低、客訴升高），而非只提供靜態報表。

### MVP 優先範圍（第一階段）

* 員工端：多日預購、廠區過濾、領餐驗證、訂單與扣款明細查詢。
* 商家端：菜單/份數管理、截單規則、備餐匯總與標籤/分區輸出。
* 福委會端：商家審核與文件到期控管、商家廠區映射、月結對帳匯出與例外處理。

### 第二階段（加值能力，不列入 MVP 驗收）

* 智慧推薦（AI / Rule-based）
* 候補通知、開賣提醒等搶購進階功能
* 商家與管理端進階營運分析儀表板
* 商家警示分級與停權/復權自動化流程

### 待驗證問題（需求凍結前必答）

* 員工實際規劃週期是 3 天、5 個工作天、還是 7 天。
* 售完資訊呈現需精準剩餘數量，或僅需「即將售完/已售完」等級即可。
* 領餐現場最可行的核銷流程與分流方式（含產線人員可行性）。
* 商家可接受的最晚截單與訂單變更時間。
* 福委會帳務異常的高頻情境（扣錯款、退款、離職員工扣款）與權責邊界。
* 第二階段推薦功能若要落地，需預先定義哪些資料來源與成功指標。

## System Requirement

以下為去重後的單一需求清單。若前文訪談摘要與本節文字重複，以本節作為實作範圍與驗收依據。

### 1. 員工

核心目標：降低選餐決策成本、提升領餐效率，並確保薪資代扣過程透明可追蹤。

* 多日預購：提供以日曆或週檢視為核心的介面，支援員工預購未來 3 至 7 天內的餐點；實際開放天數為待驗證項目。
* 廠區與可送達篩選：依員工所屬廠區與商家服務範圍自動過濾可訂餐廳，避免看到不可配送選項。
* 菜單瀏覽與篩選：支援菜單圖片、價格、類型、健康標籤與是否可送達等資訊呈現，並提供搜尋與篩選能力；此能力優先於推薦引擎。
* 薪資代扣與費用透明：整合 HR/薪資扣款流程，讓員工可查詢每筆訂單、扣款明細、退款狀態與異常申訴結果。
* 領餐驗證與核銷：支援單一且可落地的現場核銷機制，例如 QR Code、員工證或報工號；需兼顧尖峰時段吞吐量與產線人員可用性。
* 訂單管理：員工可於截單前修改或取消訂單，並查看歷史訂單、當前狀態與售完/缺貨通知。

### 2. 商家

核心目標：降低上架與備餐出錯率，穩定掌握供應量、截單節奏與多廠區配送資訊。

* 菜單與供應量管理：商家可維護菜單名稱、描述、價格、圖片、每日可供應份數，並支援菜單複製與臨時缺貨處理。
* 截單與異動規則：系統需支援每家商家設定截單時間與最晚可接受的訂單異動時間，避免出餐與配送節奏失控。
* 備餐與配送輸出：自動產生每日總表、廠區分區表、餐點標籤與配送籃清單，降低漏餐、送錯與分裝錯誤風險。
* 訂單看板：提供即時看板顯示各廠區預訂數、特殊需求、配送狀態與異常事件，方便商家掌握當日作業。
* 對帳檢視：提供月度訂單與請款摘要，讓商家可對照平台統計結果與公司結算資料。

### 3. 福委會

核心目標：確保商家合規、控制財務流程，並對服務品質異常具備治理能力。

* 商家入駐與文件生命週期：支援商家申請、文件上傳、審核、補件、到期提醒、停權與復權，並保留完整操作留痕。
* 商家與廠區映射管理：福委會可設定商家可服務的廠區與時段，並快速調整顯示權限，避免錯誤曝光不可配送店家。
* 月結扣款與例外處理：支援批次匯出符合 HR 格式的薪資代扣資料，包含鎖帳、退款、扣款失敗、離職員工等例外清單處理。
* 異常預警與治理：針對文件到期、準時率下降、滿意度偏低、客訴升高等情境提供規則化預警與後續治理流程。
* 稽核與查詢：管理端需可查詢關鍵操作、審核歷程與帳務調整紀錄，以支援稽核與責任歸屬。

## Evaluation Criteria

* **需求轉換與範圍控制**：評估是否正確實作 MVP，並將推薦、進階分析等第二階段能力維持在非核心範圍
* **程式碼品質**：代碼的整潔度、易讀性、註解完整性，並確保專案結構符合開發規範
* **架構設計與可擴展性**：模組化程度、是否易於擴充新功能（像是增加評價系統）、資料模型設計
* **系統測試與驗證**：檢查是否包含單元測試、整合測試，系統運行正確性
* **運維與可靠性**：系統穩定性、錯誤處理機制及部署後的自動化與監控能力

## Technical Constraints

* 語言偏好：Rust / SvelteKit / Tailwind CSS / PNPM / Vite Plus
* 部署環境：Kubernetes
* 可觀測性：VictoriaMetrics / VictoriaLogs / VictoriaTraces / OpenTelemetry / Grafana
* 硬體限制：盡量使用手機等現有硬體，不新增專用設備，但保留未來擴充感應器等設備的彈性
